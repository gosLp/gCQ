# Wait-free algorithms for multicore CPUs and what they suggest for 64-bit-CAS GPUs

## Executive summary

Wait-freedom on commodity multicore CPUs is now being pursued through three complementary lines of work: (i) *fast-path/slow-path* queue designs that keep the common case as scalable as the best lock-free algorithms while still forcing termination; (ii) *asymptotically improved* wait-free queues that reduce the worst-case “CAS retry” costs to polylogarithmic bounds; and (iii) *foundational support* (notably wait-free memory reclamation) required so that a “wait-free data structure” is not rendered blocking by memory exhaustion. citeturn22view0turn40view0

For GPU developers constrained to **64-bit CAS** (no 128-bit CAS / CAS2), CPU work is most directly useful when it (a) relies only on **single-word CAS + fetch-and-add/increment + swap/exchange**, (b) offers a clear path to **bounded memory** (static ring buffers, preallocation, or bounded reclamation), and (c) expresses helping/coordination in a way that could map onto GPU hierarchies (warp/wavefront → block → device). citeturn21view2turn12view0turn10view0turn11view0turn40view0

Among queue-centric designs, the most relevant comparison set for your constraints is: **SCQ (“sCQ”)** as a scalable lock-free ring-buffer baseline using only single-width atomics; **wCQ** as a wait-free extension that achieves bounded memory but (in its main form) uses **CAS2**; **LPRQ** as a high-performance LCRQ-family variant that *eliminates CAS2* using a token/“lock the cell” technique; and newer theoretical queues (OPODIS’22; PODC’23) that achieve **polylogarithmic** step complexity with **single-word CAS**. citeturn8view0turn22view0turn12view0turn10view0turn11view0

Finally, recent work on *scalable fetch-and-add itself* (PPoPP’25) and on *wait-free memory reclamation* (PPoPP’20; DISC’21 BA; PLDI’24 talk material) matters for GPUs because global atomics and safe reclamation often become the hidden bottlenecks or correctness blockers once you try to enforce wait-freedom at scale. citeturn37view0turn40view0turn41view0turn34view1

**wCQ: A Fast Wait-Free Queue with Bounded Memory Usage** (SPAA 2022; and a preliminary PPoPP 2022 poster). Authors: Ruslan Nikolaev; entity["people","Binoy Ravindran","computer engineering professor"]. citeturn22view0turn39view0  
**Abstract and contribution (summarised).** wCQ extends SCQ to provide **linearizable wait-freedom** while keeping performance close to SCQ, and it is explicitly positioned as the first fast wait-free queue with **bounded memory usage**. The SPAA paper argues that “true wait-freedom” implies bounded memory usage, because unbounded memory usage makes the system blocking when memory is exhausted. citeturn20view2turn22view0  
**Algorithm sketch.** wCQ uses a **fast path** that is essentially SCQ and a **slow path** that guarantees termination. Each operation tries the fast path for a **bounded number of attempts** (`MAX_PATIENCE`), and if it fails it publishes a help request in a per-thread record and enters a cooperative slow procedure that uses a specialised “slow_F&A” technique to coordinate head/tail advancement and converge helpers on a single stalled thread. citeturn21view2turn22view0turn39view0  
**Progress guarantee.** The SPAA paper proves wCQ’s operations are **wait-free** (Theorem 5.10), and it separately proves **bounded memory usage** (Theorem 5.8) by noting it is statically allocated and uses only bounded per-thread descriptors. citeturn22view0  
**Hardware assumptions.** wCQ leverages fetch-and-add for scalability and requires **double-width CAS (CAS2/WCAS)** for correctness of slow-path procedures on mainstream x86-64 and AArch64; it also illustrates an implementation strategy for architectures that lack CAS2 by building CAS2 behaviour via LL/SC. citeturn20view2turn22view0turn39view0  
**Memory model.** wCQ assumes **sequential consistency for presentation** and states that the implementation inserts barriers as needed. citeturn20view2  
**Complexity.** The fast path is bounded by `MAX_PATIENCE` attempts before falling back, and the slow-path coordination is proved “wait-free bounded” via convergence arguments (e.g., Lemma 5.5), though the paper does not package this as a simple closed-form in n (treat the exact numeric step bound as **unspecified**; what is specified is boundedness plus the `MAX_PATIENCE` gating). citeturn22view0turn21view2  
**Implementation / code.** The SPAA paper states code availability and points to an official repository; the companion repository README indicates wCQ is implemented using double-width CAS and is provided via a submodule arrangement. citeturn20view2turn29view1turn29view0  
**GPU applicability (64-bit CAS).** Medium *conceptually*, low *directly*: the design pattern (bounded fast-path retries + cooperative slow-path helping while keeping memory bounded) is highly relevant to GPU scheduling, but the reliance on **CAS2** makes a direct port infeasible under a 64-bit-CAS-only constraint. The PPoPP poster explicitly notes wCQ uses double-width CAS and achieves wait-free progress by avoiding dynamic allocation/memory reclamation in the queue itself—both are good principles for GPUs, but the CAS2 dependence is the key blockade. citeturn39view0turn22view0turn12view0  
**Likely adaptation direction for GPUs.** The most promising CPU-derived adaptation is to replace CAS2-dependent per-cell epoch/state updates with a **single-word CAS protocol** (token/indirection packing) akin to LPRQ’s elimination of CAS2, at the cost of extra steps and careful ABA/epoch handling. (This is an inference supported by the fact that LPRQ explicitly removes CAS2 while preserving LCRQ-style epochs, and wCQ’s poster highlights epoch/state as the thorny part of F&A ring buffers.) citeturn12view0turn39view0turn22view0

**The State-of-the-Art LCRQ Concurrent Queue Algorithm Does NOT Require CAS2** (PPoPP 2023; Feb–Mar 2023 proceedings). Authors: entity["people","Raed Romanov","concurrency researcher"] and entity["people","Nikita Koval","software engineer and researcher"]. citeturn12view0turn13view0  
**Abstract and contribution (summarised).** This paper introduces **LPRQ**, a portable modification of LCRQ that **eliminates CAS2** and uses only standard **CAS + fetch-and-add**, reporting performance comparable to LCRQ and up to **1.6×** better than other “no CAS2” alternatives. citeturn12view0  
**Algorithm sketch.** LCRQ uses per-cell epochs updated together with cell state via CAS2; LPRQ removes CAS2 by shifting epoch responsibility primarily to enqueues and using **thread-unique tokens** to “lock” a cell: an enqueue installs a token, checks epoch correctness, then replaces the token with the element. Tokens can be implemented by bit-stealing/tagging or (on managed runtimes) by thread references. citeturn12view0  
**Progress guarantee.** The paper proves the base PRQ ring buffer is **obstruction-free** (Theorem 4.3) and notes how to upgrade to **lock-freedom** by closing the ring buffer after a constant number of unsuccessful attempts, following LCRQ/SCQ-style approaches (the paper states the lock-free proof is omitted, referencing prior work). citeturn13view0  
**Hardware assumptions.** Only **single-word CAS + fetch-and-add** are required in the LPRQ design; CAS2 is discussed as non-portable and is explicitly avoided in the final algorithm. citeturn12view0  
**Memory model.** The paper assumes **sequential consistency** for simplicity and uses CAS/FAA primitives; it also assumes a garbage-collected environment for simplicity but mentions hazard pointers/eras as options for manual reclamation. citeturn12view0  
**Complexity.** The paper is performance-driven and does not present polylog step-complexity style theorems (treat asymptotic step complexity bounds as **unspecified** in that formal sense); it does, however, describe bounded-attempt modifications for lock-freedom and provides experimental comparisons. citeturn13view0turn12view0  
**Implementation / code.** The paper states a C++ implementation and includes an evaluation artefact reference; independent implementations exist, such as a Rust implementation explicitly positioning itself as an LPRQ port. citeturn13view0turn29view3  
**GPU applicability (64-bit CAS).** High for the specific requirement “avoid CAS2”: LPRQ is one of the clearest demonstrations that a high-performance FAA-centric ring-buffer queue can be made **portable to single-word CAS** by transforming the epoch/state update into a **token + validation** protocol. The limitation is that LPRQ is **not wait-free** (obstruction-free/lock-free depending on configuration), so it does not solve per-thread bounded latency by itself; it is best viewed as a **CAS2-removal technique** that could be combined with a separate wait-free guaranteeing outer layer (fast/slow path) if the GPU use case needs strict wait-freedom. citeturn13view0turn12view0turn22view0

## Supporting techniques for practical wait-freedom

**Universal Wait-Free Memory Reclamation** (PPoPP 2020). Authors: Ruslan Nikolaev and Binoy Ravindran. citeturn40view0turn31view0  
**Abstract and contribution (summarised).** The paper introduces **Wait-Free Eras (WFE)**, a *universal* memory reclamation scheme for deleted blocks in wait-free concurrent data structures, with the key claim that the reclamation operations themselves are **completely wait-free**. It argues that pairing a wait-free data structure with only lock-free hazard pointers or potentially blocking epoch-based schemes effectively forfeits the data structure’s wait-free guarantee. citeturn40view0  
**Algorithm sketch.** WFE extends the **Hazard Eras** scheme to guarantee wait-freedom and discusses the challenges in making a universal reclamation approach wait-free. WFE reviews the hazard-eras API shape (get_protected/retire/clear/alloc_block) and how eras and reservations bound memory usage. citeturn40view0  
**Progress guarantee.** WFE’s core claim is **wait-free progress** for reclamation operations, enabling “fully wait-free” implementations of wait-free data structures that otherwise needed weaker reclamation. citeturn40view0  
**Hardware assumptions.** The paper explicitly states WFE is implementable on x86_64 and AArch64, and it discusses the use of native **fetch-and-add** plus **wide CAS (WCAS)** on those platforms, distinguishing WCAS from double-CAS. citeturn40view0  
**Memory model.** The paper is written in the usual atomic-operation shared-memory model; it does not pin itself to a specific language memory model in the extracted portions (treat as **unspecified** at the C++11/CUDA level). citeturn40view0  
**Complexity.** The abstract emphasises performance close to epoch-based reclamation and hazard eras while providing stronger progress; exact asymptotic step bounds are not summarised in the abstract (treat formal step-complexity expressions as **unspecified** here). citeturn40view0  
**Implementation / code.** The authors provide official code in a dedicated repository and note that the API is mostly compatible with hazard pointers. citeturn40view0turn31view0  
**GPU applicability (64-bit CAS).** Mixed: WFE explicitly relies on **WCAS** in its atomic-operations discussion, which clashes with a 64-bit-CAS-only constraint; however, the *conceptual contract*—memory usage must be bounded and reclamation itself must not introduce blocking—maps directly to GPU design. In many GPU designs, you may sidestep the hardest reclamation issue by using **preallocated pools** and deferring reclamation to kernel boundaries, but if you need long-lived device-resident structures, WFE’s architecture is a valuable reference for what must be made wait-free. citeturn40view0turn20view2turn41view0

**Abstract and contribution (summarised).** The brief announcement presents **Crystalline**, claiming **complete wait-freedom** even with dynamically recycled threads, **asynchronous reclamation** (any thread can reclaim another’s retired memory), and (almost) **balanced reclamation workload** across threads—argued to yield high performance and memory efficiency. citeturn41view0

**Progress guarantee.** The BA proposes **Crystalline-W** as wait-free and contrasts it with a lock-free Crystalline-L variant. citeturn41view0turn34view1  
**Hardware assumptions.** The BA and associated slides emphasise reliance on unconditional primitives like **SWAP** and on architectures where fetch-and-add and swap are available, plus wide-CAS-like support in the broader family talk materials; constraints are hardware-oriented rather than language-oriented. citeturn41view0turn34view1

**Complexity.** The BA is performance-focused (scalability and memory efficiency as threads increase) rather than giving closed-form step complexity bounds (treat as **unspecified**). citeturn41view0  
**Implementation / code.** The BA lists official source code as supplementary material and the associated repository README states the Crystalline code is embedded in benchmarking infrastructure. citeturn41view0turn31view1  
**GPU applicability (64-bit CAS).** High as a *design checklist*: Crystalline’s stated focus on eliminating non-converging loops (CAS-loop nontermination under contention) and making “protect” bounded is exactly the kind of issue that arises when GPUs oversubscribe a shared structure. Still, the described reliance on SWAP and era mechanisms implies heavy atomic traffic and metadata, which may be too costly device-wide; a more plausible GPU mapping is **per-block reclamation discipline** or host-assisted reclamation, using Crystalline/WFE ideas to understand which loops must be bounded and by what coordination. citeturn41view0turn40view0turn34view1

**Abstract and contribution (summarised).** L-UC is a **wait-free universal construction** for dynamic objects with large state, with step complexity **O(n + k·w)** where k is interval contention and w is the cost of the sequential operation. It targets “large objects” by avoiding whole-state copying and by batching operations via a wait-free analogue of combining. citeturn24view1  
**Algorithm sketch.** L-UC uses an **Announce** array for operations and a **Toggles** bit-vector mechanism to signal pending operations; processes coordinate in phases and apply batches of requests, using a local “directory” (cache) of accessed shared variables so that updates can be applied efficiently without copying the full object. citeturn24view1turn26view0  
**Progress guarantee.** Wait-free, by construction and stated in the abstract. citeturn24view1  
**Hardware assumptions.** The OPODIS write-up presents L-UC using **LL/SC objects** but explicitly states a practical version would use **CAS objects**; it also notes that some fields would require atomic updates of **two words** (impractical in small-register hardware) but that **indirection** can reduce this to single-word registers, including for the main shared state variable S. citeturn25view1turn26view0  
**Memory model.** Not specified beyond the base-object model (treat as **unspecified** at language level). citeturn24view1  
**Complexity.** O(n + k·w) step complexity is explicitly stated. citeturn24view1  
**Implementation / code.** No official implementation is indicated in the retrieved material (treat as **unspecified**). citeturn24view1turn25view0  
**GPU applicability (64-bit CAS).** Useful mainly as *architecture inspiration*, not as a direct template: universal constructions are powerful but tend to be heavy. The points most transferable to GPUs are (i) **batching/combining** as a way to amortise synchronisation and (ii) using **indirection** to avoid multiword atomic updates. Both map naturally to GPU block-level delegation (one “combiner” thread applies a batch) and to handle-based memory representations. citeturn24view1turn26view0turn37view0

**Aggregating Funnels for Faster Fetch&Add and Queues** (PPoPP 2025; March 2025 proceedings). Authors include Younghun Roh, Yuanhao Wei, Eric Ruppert, Panagiota Fatourou, entity["people","Siddhartha Jayanti","computer scientist"], and entity["people","Julian Shun","computer scientist"]. citeturn37view0  
**Abstract and contribution (summarised).** This paper targets the scalability bottleneck of contended fetch-and-add by proposing **Aggregating Funnels**, claiming they outperform hardware F&A beyond ~30 threads and improve LCRQ throughput by up to **2.5×** when replacing the contended F&A objects in the queue. It also states their fetch-and-add implementation is **strongly linearizable**, positioning it for safe composition. citeturn35view0turn37view0  
**Relevance to wait-freedom.** While not itself a wait-free queue paper, it is directly relevant because many “fast” non-blocking queues rely on FAA for head/tail coordination (including LCRQ/SCQ/LPRQ/wCQ), and the practical ceiling often becomes atomic contention rather than algorithmic step bounds. citeturn35view0turn36view1turn20view2  
**Hardware assumptions.** The approach uses hardware fetch-and-add internally and is evaluated across CPU platforms; it reports cross-platform behaviour and explicitly mentions AMD EPYC and Intel Xeon families in experiments. citeturn37view0  
**GPU applicability (64-bit CAS).** Conceptually high: GPUs often suffer from precisely the same “contended atomic” bottleneck. The architectural pattern—**hierarchical aggregation plus delegation** to reduce contention on a single memory location—resembles common GPU optimisation patterns (warp/block aggregation). The caveat is that the paper’s mechanism is designed for CPU cache-coherence behaviour; mapping it to GPU memory systems would require redesigning the aggregation layer around GPU execution groups and memory scopes. citeturn35view0turn37view0

**Nontrivial and universal helping for wait-free queues and stacks** (Journal of Parallel and Distributed Computing, November 2018). Authors: entity["people","Hagit Attiya","computer scientist"], entity["people","Armando Castañeda","computer scientist"], entity["people","Danny Hendler","computer scientist"]. citeturn27view0  
**Contribution (summarised).** This work formalises “helping” in wait-free implementations via (i) **nontrivial helping** based on operation valency and (ii) **universal helping** linked to restricting linearisation possibilities, and it argues (among other results) that wait-free queue implementations from certain weaker primitives require nontrivial helping, while universal helping has consensus-strength consequences. citeturn27view0  
**GPU applicability.** As theory, it does not yield a drop-in algorithm, but it clarifies that to obtain wait-freedom you almost inevitably need **helping mechanisms** that can “decide” outcomes for other threads—exactly the style seen in wCQ’s cooperative slow path and in universal constructions. This supports a GPU strategy of explicit, hierarchical helping (warp-level helping, then block-level). citeturn27view0turn22view0turn24view1

## GPU applicability synthesis and adaptation patterns

The CPU literature suggests that **64-bit-CAS GPUs** should prioritise designs that avoid multiword atomics by *construction*, not by emulation. LPRQ’s core contribution is a concrete example: remove CAS2 by using a **token + validate + publish** protocol so that each “atomic multi-field update” is decomposed into a sequence of single-word CAS operations with well-defined intermediate states. citeturn12view0turn13view0

For **bounded memory** (which is crucial for practical wait-freedom), wCQ’s approach is the most GPU-aligned: it explicitly avoids dynamic allocation and proves bounded memory usage by being a statically allocated circular queue with bounded per-thread descriptors. This is a strong match for GPU kernels that already prefer preallocated pools and fixed-size buffers. citeturn22view0turn39view0turn40view0

For **enforcing termination**, the strongest reusable pattern is **bounded fast-path retries followed by cooperative helping**. wCQ operationalises this directly (`MAX_PATIENCE` gating and a cooperative slow path), and Crystalline’s reclamation work shows the same idea is needed in memory management: remove non-converging CAS loops (e.g., by switching to unconditional SWAP or by forcing slow-path coordination when “protect” loops fail). citeturn21view2turn22view0turn41view0turn34view1

For GPU-scale contention, the theoretical queues (OPODIS’22; PODC’23) and the scalable-FAA work (PPoPP’25) point to a likely reality: you will often need a **hierarchical design**. Use something SCQ-like (simple ring buffer, single-word atomics) at the lowest level, and combine it with (i) a polylog-coordination mechanism for cross-group fairness/termination, or (ii) aggregation layers that reduce global atomic contention (akin to funnels/combining). citeturn8view0turn10view0turn11view0turn35view0turn37view0

A concrete, GPU-oriented blueprint implied by these sources is:

- **Intra-block queue**: SCQ-style ring buffer (single-word CAS + FAA + swap) for throughput and bounded memory, possibly with a wCQ-like bounded retry and block-level helping for tighter latency bounds. citeturn8view0turn39view0turn22view0  
- **Inter-block distribution**: either tolerate weaker progress (lock-free) with LPRQ-style single-word CAS portability, or introduce additional coordination (tree/combining) inspired by OPODIS’22/PODC’23 but parameterised for the smaller “n” at the block/SM granularity. citeturn12view0turn10view0turn11view0  
- **Memory discipline**: if you must reclaim device-side nodes, treat WFE/Crystalline as guidance that reclamation must be bounded and non-blocking; otherwise, prefer preallocation and reclamation at kernel boundaries (not a claim from the papers, but a common GPU engineering choice consistent with their “bounded memory to preserve progress” requirement). citeturn40view0turn41view0turn20view2
